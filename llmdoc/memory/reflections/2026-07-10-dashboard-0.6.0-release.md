# 反思:Dashboard 0.6.0 整站大修与发布

## Task

- 用户在查看上一版生产截图后明确反馈“变化不是很大”,随后允许大修,并把唯一偏好收敛为:树要明显更好,但登录、总览、Node、Context 和全部 system 页面也必须同时兼具功能性与美观性。
- 在不改变 HTBP、权限、builtin、CLI 和宿主契约的前提下,把 Dashboard 从“单侧栏 + 纵向卡片”重组为 ActivityRail + Resource Explorer + 全宽 Workspace,并让树、命令、Context 和管理工作流使用同一套产品层级。
- 保留 0.5.0 已建立的敏感数据、cursor、乐观并发、lazy chunk 和远端树性能边界,修复大修新增的 observer 生命周期、并发操作和 ARIA tree 风险。
- 完成 `@tool-bridge/dashboard` 0.6.0 的 npm 发布与生产 `/ui` Static Assets 上线,分别验证两个发布面。

## Expected vs Actual

- 预期:用户最在意树,把树做得更清晰、更像资源浏览器,再统一一些色彩和间距就能形成明显升级。
- 实际:“变化不大”不是要求再换一轮视觉 token,而是在指出产品结构仍像旧版。只有 ActivityRail、Explorer、Workspace 的信息架构和所有路由一起变化,用户才会把它感知为整站大修;只重做树会让其他页面成为割裂的旧界面。
- 预期:视觉重组主要影响布局组件。
- 实际:只挂载当前命令、同页允许多行操作、移动端改用 Dialog/Drawer 后,React observer 与视图的挂载周期发生了变化。最危险的问题来自业务操作被错误绑定到 MutationObserver 生命周期,不是 CSS。
- 预期:通过源码审查、typecheck 和 build 后即可发布。
- 实际:发布前审计先发现 version 仍为 0.5.0、全部实现尚未提交;即使代码可发布,此时打 `dashboard-v0.6.0` 也会漏掉大修或被 workflow 的版本校验拒绝。代码质量与 Git/版本状态必须分别裁决。

## What Went Wrong

1. **第一轮优化停留在局部精修。** 保留旧信息架构时,字体、边框、卡片和色彩即使更统一,第一眼仍是同一个产品。用户反馈证明视觉验收要从页面结构和任务流开始,不能把“好看”缩成样式变量。
2. **把调用级 mutation callback 当成操作生命周期。** Plugin 多行探活/启停共用 MutationObserver 时,连续 `mutate` 可能替换前一调用的 observer callback,导致某一行永久停在 probing/pending。类似地,当前 CmdPanel 被切换、Tab 或路由卸载后,挂在 observer 上的成功失效逻辑可能消失。
3. **只锁本地切换不足以保证正确性。** pending 时禁止切换命令能避免一次常见 remount,却挡不住路由、Tab 或整个页面卸载。交互锁是 UX 保护,不能成为后台操作结算的唯一保障。
4. **树的视觉控件与键盘模型曾有两个状态所有者。** 展开按钮和节点链接都进入 Tab 序列时,一行会出现多个 tab stop;方向键只在部分元素上工作,展开状态、可见顺序和焦点也可能不同步。
5. **把已加载页中的“不存在”误写成全局不存在。** Secret `set` 是 upsert,Registry `write` 是同 path 整体替换;cursor 尚有下一页时,当前已加载 items 找不到名称/path,只能说明“未知”,不能承诺“新建”。
6. **专用页面的视觉扩写容易掩盖能力漂移。** Registry、Plugin、SK 等大表单若只按截图迁移,容易漏 OAuth 分支、NodeConfig 字段、一次性凭据或危险操作语义。好看的专用页面仍必须接受 builtin/store、CLI、Dashboard 三向对等审查。
7. **“发布”再次暴露为两个独立状态。** npm tag workflow 成功不代表生产 `/ui` 已更新;Cloudflare Worker 已自动部署时再手工 deploy,也只会制造重复 version。两条发布面不能用同一个版本字段或一次写操作替代验证。

## Root Cause

- 把视觉设计理解为组件皮肤,而不是全局导航、资源发现、任务上下文和页面层级共同形成的信息架构。
- 把 React 组件和 TanStack MutationObserver 当成业务操作所有者;实际上网络请求、toast、局部清理、失败重抛和 query invalidation 的生命周期可能长于当前视图。
- 把 cursor 分页当作加载体验,没有把它视为知识边界:只加载部分集合时,前端不能证明名称或 path 在全集中不存在。
- 树的协议查询状态、展开状态、筛选结果、可见顺序和键盘焦点分散拥有,导致补 ARIA 时容易额外触发查询或产生双 tab stop。
- 将 Dashboard package version、Gateway runtime version、Worker deployment version 与 Static Assets 身份笼统称为“线上版本”,模糊了每个证据真正证明的对象。

## What Worked

1. **把用户的负反馈升级为整站信息架构约束。** ActivityRail 保持全局入口稳定,Explorer 专注资源树和路径上下文,Workspace 承载当前任务;登录、总览、Node、Context 与六个 system 页面都迁入同一视觉语法,而不是只给树换皮。
2. **视觉审查与契约审查并行。** 一条复核布局、树、工作台、移动交互和键盘模型;另一条逐项核对 builtin schema/store、CLI 与 Dashboard 字段。最终独立审查未发现遗留 P0/P1/P2,同时保住了美观与功能边界。
3. **必要副作用进入独立 Promise 控制流。** 可并发或可卸载的操作使用 `mutateAsync`,并在函数自身完成 toast、状态清理、失败重抛和 query invalidation。组件是否仍挂载不再决定操作是否结算。
4. **树用一个受控模型连接协议与无障碍。** `expandedPaths`、可见节点顺序和 roving tab stop 共同承载展开与焦点;节点行只有一个 Tab 入口,方向键、Home/End、Space/Enter 遵循 ARIA tree 语义,且键盘移动不会偷偷扩大查询深度。
5. **对部分集合保持诚实。** 页面统一使用“已加载”口径;Secret/Registry 在还有后续页时,把未知同名/同 path 明确提示为潜在轮换或整体替换,没有把 upsert 伪装成纯 create。
6. **真实浏览器证据覆盖了静态审查看不到的面。** 本地生产构建在桌面与移动两类视口巡检登录、总览、Node、Context、Registry、SK、Secret、Devices、Plugin、Federation,无页面级横向溢出,console 为 0 error / 0 warning。网络实测首屏只取 root `depth=1`,展开本地分支才取该分支 `depth=1`,非空筛选才取 root `depth=8`;roving tab stop 与 ArrowDown 也已实测。
7. **敏感值与语义用真实交互复核。** 浏览器实测 `system/sk list` 成功、Context 桌面主从与移动详情 Dialog 可用、SK 一次性明文不能由 Escape 关闭且必须显式确认、Secret 保存后不回显并对同名轮换告警、Registry 对同 path 整体替换告警。
8. **先读平台状态再决定部署。** `main` 推送已由 Cloudflare 集成生成目标 Worker version,因此没有为获得命令输出而重复执行 `deploy:all`;npm 和生产静态资产再用各自证据收口。

## Stable Design Boundaries

### 树与 App Shell

- Tree-first 不等于 Tree-only。ActivityRail 拥有稳定的全局/管理入口,Explorer 拥有资源筛选、路径与树,Workspace 不因 Explorer 折叠而卸载;移动端可以换成抽屉,但任务上下文和关闭后的焦点恢复不能丢。
- 树的性能边界保持为:常态 root `depth=1`,本地 truncated 分支按 `depth=1` 懒加载,remote 根及其后代按纯透传 `depth=3` 懒加载,只有非空筛选启用单条 root `depth=8` 查询。remote 返回路径必须 localize 回本地挂载前缀;direct URL 不应静默触发深查询。
- ARIA tree 不是补几个 role。展开、筛选、可见顺序、选中态和 roving focus 应共享状态设计;每行一个 tab stop,方向/Home/End/Space/Enter 的行为与视觉展开一致,且不得为了焦点导航改变网络查询边界。

### Node、命令与 Context 工作台

- NodePage 继续是未知 HTBP 命令的通用协议兜底;专用页面可以优化已知工作流,但不能替代协议驱动的 NodePage。
- CommandWorkspace 只挂载当前 CmdPanel,只为当前 schema 请求工具级 `~help`;pending 时锁筛选/切换用于减少误操作,但调用与变更型失效必须由跨卸载安全的 Promise 链持有。
- Context 桌面主从和移动详情可以采用不同呈现,数据边界不变:List/Search/cursor、`$ref`、metadata 均保留;正文、version、`$ref` 从同一次 Get 捕获原子编辑基线,写入携带 `ifVersion`,大对象只经 Update 修改 metadata。

### 全部 system 页面

- SK、Secret、Registry、Devices、Plugin、Federation 可以使用各自最合适的宽表、筛选、详情和创建工作流,但必须共享页面层级、错误/空态、分页和危险确认原语。
- 专用页面的功能真源始终是 builtin schema/store 与 CLI 命令矩阵。Registry 保留全部 NodeConfig/OAuth/virtualize 分支;Plugin 保留六动作;SK 保留 scope 全字段;Devices 不承诺后端不存在的递归删除;Federation 的 env 基线保持只读。
- 多行并发、危险操作和一次性凭据不能依赖共享 observer 的最后一次 callback。Secret value、SK secret、pluginToken 的展示、关闭与 reset 规则属于安全边界,不是可由视觉重排省略的交互细节。
- 有 cursor 时所有计数、筛选、选择器和冲突提示只对“已加载集合”负责;create、upsert、replace 必须按后端真实语义命名。

## Release Boundary

- Dashboard npm 发布面由 `dashboard-v<version>`、publish Actions 和 npm dist-tag 共同证明;生产 `/ui` 发布面由 Cloudflare deployment/Worker version、HTML 与入口资源身份、SPA 深链接和 smoke 共同证明。
- 推送 `main` 后先查 Cloudflare deployments/versions 和生产产物。目标静态资产已经自动上线时停止,不要重复手工部署。
- `/healthz.version` 只属于 Gateway runtime,不能证明 Dashboard npm 或 `/ui` Static Assets 版本。报告版本时必须写明所有者。
- tag 前先证明 version bump、完整实现提交和目标 commit 同在;代码审查通过不等于工作区已具备可打 tag 的 Git 状态。

## Missing Docs or Signals

- 本轮真实浏览器验收仍是一次性手工证据,产品级可重跑回归尚未覆盖树查询深度、ARIA 键盘、跨卸载 mutation/invalidation、Plugin 多行并发、pending Dialog 与 cursor 冲突提示。
- Plugin 并发、跨 Tab/路由卸载后的 mutation 结算以及 Context pending 时防关闭,本轮依据是代码审查、typecheck/build 与 Promise 控制流复核,不能误写成已完成的浏览器实测。
- 大型 system 页面仍同时承担查询编排、config builder、表单状态和视图;它不是 0.6.0 发布 blocker,但继续加字段前应拆出纯 builder 和共用 section,降低三向对等漂移。

## Stable Promotion Candidates

1. **操作生命周期独立于 observer 生命周期:**任何允许并发 mutate 或可能卸载 mutation owner 的界面,必要副作用都应由独立 `mutateAsync` Promise 链或按操作隔离的 observer 持有;禁止只依赖调用级 callback。
2. **局部 pending 锁不承担正确性:**禁止切换/关闭只改善当前交互,后台操作在没有视图时仍须完成结算、失败传播和缓存失效。
3. **分页是知识边界:**当 `hasNextPage` 为真,已加载集合中的缺失不能证明全局缺失;upsert/replace UI 必须提示潜在同名或同 path 冲突。
4. **树的五态共同设计:**协议懒加载、展开、筛选、可见顺序与 ARIA 焦点必须一起设计和验收;无障碍增强不得破坏远端树的性能边界。
5. **整站视觉大修要做能力矩阵:**页面结构变化必须覆盖所有路由族,并在迁移前后核对 builtin/store、CLI、Dashboard,避免视觉完整而功能缩水。
6. **发布面先读后写:**npm 与生产 Static Assets 分开列状态、分开取证;先查自动部署,只在目标状态缺失时执行外部写操作。
