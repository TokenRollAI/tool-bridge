# 反思:Dashboard 0.5.0 发布与生产静态资产验收

## Task

- 清理本轮前端重构留下的 Playwright 会话、QA 截图与临时数据,只保留代码和可复用知识。
- 将 Dashboard 版本升至 0.5.0,按职责拆成四笔 pathspec 提交并 fast-forward 推送 `main`。
- 通过单独推送 `dashboard-v0.5.0` tag 触发 npm Trusted Publishing,并验证 registry 的 `latest` 已是 0.5.0。
- 确认 `main` 推送触发的 Cloudflare 自动部署已经把同一份 Dashboard 静态产物送到生产,再做产物 hash、资源可达性和 smoke 验收。

## Expected vs Actual

- 预期:清理临时产物、推送代码、发 tag、等 npm workflow 结束,然后显式部署生产。
- 实际:本轮有两个彼此独立的发布面。`dashboard-v0.5.0` 只负责 npm 包发布;生产 `/ui/` 属于 Gateway 的 Static Assets,由 `main` 推送后的 Cloudflare 集成自动部署。仓库外已生成 Worker version 43(ID 前缀 `dc380d92`),若再运行 `pnpm deploy:all` 只是重复部署,不会增加有效覆盖。
- 内置 Browser 当时不可用,无法由 Agent 再做一轮浏览器交互;但用户截图确认了生产视觉,同时生产 HTML 与本地构建的 SHA-256 完全一致、关键 chunk 均返回 200、深链接回退正确且 smoke 通过,因此发布证据并未依赖单一工具。

## What Went Wrong

1. **起初把“发版”理解成一个动作。** npm 的 tag workflow 与 Cloudflare 的生产静态资产部署是两条独立流水线。只看其中一个成功,无法证明另一个发布面已经就绪;反过来,两条都已经成功时继续手动 deploy 会制造无意义的新 Worker version。
2. **容易把 Gateway 的运行时版本误认成 Dashboard 包版本。** `/healthz` 中的 0.4.0 描述 Gateway,不是 `/ui/` 所承载的 Dashboard npm 版本。若只看健康接口的版本字段,会错误判断前端仍未发布。
3. **清理范围需要先分类。** `.playwright-cli/`、`output/playwright/`、QA PNG 和临时本地数据都不是知识真源,可以删除;用户提供的截图、源码资产、发布产物及 llmdoc 不能因为扩展名或目录名相近而批量清理。
4. **网络抖动会伪装成发布失败。** push/远端查询期间偶发 `SSL_ERROR_SYSCALL`;重试后成功,说明它是传输层瞬时故障,不应据此改凭据、重建 tag 或重复部署。

## Root Cause

- 发布流程文档分别描述 npm 与生产部署,但任务语言“发版”天然会把版本提交、registry 发布、静态资产上线和视觉确认压缩成一个结果。执行时若不先列发布面,就会把某条流水线的状态外推到另一条。
- Static Assets 的真实版本不由 `/healthz.version` 表达;Gateway 版本、Dashboard package version、Worker version 和带 hash 的静态 chunk 各自回答不同问题。缺少产物级比对时,版本字段很容易被错误关联。
- QA 清理如果只按文件类型做判断,会混淆“临时浏览器产物”和“用户输入/项目资产”。安全边界应由来源、生命周期及是否被 Git 跟踪共同决定。
- 对远端动作缺少“先读当前状态再写”的习惯时,网络错误和异步平台延迟都可能诱发重复发布。

## What Worked

1. **四笔 pathspec 提交保持边界清晰。** 前端实现、临时产物忽略规则、llmdoc 更新和 0.5.0 版本提升分别提交,随后 fast-forward 推送 `main`;最终 `HEAD`、`origin/main` 与发布 tag 指向同一版本提交,便于审计和回滚。
2. **tag 单独推送。** `dashboard-v0.5.0` 单独 push 后,GitHub Actions run `29046855598` 成功;`npm view @tool-bridge/dashboard dist-tags.latest` 返回 0.5.0,同时覆盖了 workflow 成功和 registry 可消费两个层次。
3. **先查自动部署再决定是否手动部署。** `main` 推送已经在仓库外生成 Cloudflare Worker version 43,因此停止重复 `deploy:all`;这既避免新版本噪声,也遵守“真实外部部署每轮只做必要一次”的纪律。
4. **用产物 hash 证明生产内容。** 生产 `/ui/` 与深链接返回的 HTML SHA-256 和本地 `dist/index.html` 完全一致,页面引用的关键 hash chunk 均为 200。它比 `/healthz` 的 Gateway 版本号更直接地证明了新 Dashboard 已上线。
5. **多证据收口。** 静态产物 hash、资源状态码、深链接回退、生产 smoke 和用户截图分别覆盖内容一致性、静态托管、SPA 路由、网关健康与视觉结果;内置 Browser 不可用没有变成发布盲点。
6. **网络故障保持幂等重试。** 对 `SSL_ERROR_SYSCALL` 只重试原读/推操作,先确认远端 refs、Actions 和 npm 状态,没有改认证配置或重建已成功对象。

## Missing Docs or Signals

- npm 发布指南需要明确提醒:Dashboard 的 npm tag 发布不等于生产 `/ui/` 上线;后者取决于承载 Static Assets 的 Gateway 部署流水线。
- 部署指南需要补充“先查平台 deployments/versions,再决定是否手动 deploy”,尤其是在仓库已配置主干自动部署时。
- Dashboard 生产验收应优先比较本地/线上 HTML 与关键 chunk hash,并明确 `/healthz.version` 是 Gateway 版本,不能作为 Dashboard 包版本证据。
- 清理指南应区分 gitignored QA 产物、用户输入附件、源码资产和正式构建产物,避免以 `.png` 或 `playwright` 关键词做宽泛删除。

## Stable Promotion Candidates

1. **先画发布面再执行:**对“发版”列出版本提交、tag/npm、生产部署和终端验证四个状态;每个状态都要有自己的读取证据,不得相互替代。
2. **读后写、避免重复部署:**推送后先查询 GitHub Actions、npm dist-tag、Cloudflare deployments/versions 和生产产物;只有目标状态缺失时才触发下一次外部写操作。
3. **静态前端用产物身份验收:**比较线上 HTML 与本地构建 SHA-256,再逐个请求入口引用的 hash chunk;SPA 还要验证深链接回退。运行时健康版本只能证明宿主,不能证明 UI 包版本。
4. **清理按所有权和生命周期:**仅删除已确认由本轮 QA 产生且可再生的未跟踪/gitignored 产物;用户附件、跟踪资产和知识文档不纳入通配清理。删除后复查 `git status` 和忽略规则。
5. **瞬时网络故障只做幂等重试:**遇到 `SSL_ERROR_SYSCALL` 等传输错误,先重试并读远端事实;不要未经证据旋转凭据、重打 tag 或重发部署。
6. **版本命名必须带所有者:**报告中使用“Dashboard npm 0.5.0”“Gateway health 0.4.0”“Worker version 43”,避免裸写“线上版本 0.4.0/0.5.0”造成错误归属。

## Task-Specific Evidence (Keep in Memory)

- 本轮发布链路为四笔 pathspec commit → fast-forward push `main` → 单独 push `dashboard-v0.5.0` → Actions run `29046855598` 成功 → npm `latest=0.5.0`。
- `main` 推送后 Cloudflare 自动生成 Worker version 43(ID 前缀 `dc380d92`);没有再执行重复部署。
- 生产 `/ui/` 与深链接 HTML 的 SHA-256 均与本地构建一致,关键 chunk 返回 200,smoke 通过;这些证据证明的是生产静态资产,不是 npm registry 本身。
- 内置 Browser 在发布验收时不可用,最终视觉由用户截图确认;这次是工具降级路径,不应替代后续可重跑浏览器回归。
- `.playwright-cli/`、QA PNG 和临时数据均为可再生验收产物,本轮已清理;以后仍应先确认 Git 状态和来源,不把所有 PNG 一概视为垃圾。
- 网络访问曾偶发 `SSL_ERROR_SYSCALL`,原操作重试后成功,没有证据表明凭据或发布配置异常。

## Follow-up

- recorder 将“发布面拆分、先查 deployments/list、静态产物 hash 验收、版本所有者命名和 QA 清理边界”提炼进 npm 发布/部署与验证指南。
- 后续 Dashboard 发布应把 npm registry 与生产 `/ui/` 分开报告:前者提供 workflow run + dist-tag,后者提供 Worker deployment/version + HTML/chunk hash + smoke/浏览器证据。
