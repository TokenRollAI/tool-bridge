# 反思：CLI 0.8.0 发布准备

## Task

- 为 CLI 参数契约大改准备 npm 新版本：确认 registry 基线、判断版本提升范围、更新版本并验证实际发布物。
- 在尚未打 tag、尚未发布的边界内，证明 `@tool-bridge/cli` 0.8.0 的构建产物和 npm tarball 可用于后续发布。

## Expected vs Actual

- 预期：全仓 `pnpm verify` 已有 **1103 passed / 7 skipped**，把 CLI 从 registry `latest=0.7.0` 升到 0.8.0 后再跑一次 build/pack 即可。
- 实际：第一次发布物验证用了错误的 `../../node_modules/.bin/tsup` 路径，而且多个 shell 命令没有用 `&&` 连接。tsup 失败后，后续 `npm pack` 仍成功并让整条命令以 0 退出；旧 `dist` 随之被打包，`node dist/index.js --version` 仍显示 0.7.0。若只看最终退出码或 pack 成功，会把陈旧产物误认成 0.8.0。
- 修正为包内 `./node_modules/.bin/tsup && node dist/index.js --version && npm pack --dry-run --json` 后，三个阶段形成失败即停的证据链：重建成功、产物报告 0.8.0、tarball 只含 `LICENSE`、`README.md`、`dist/index.js`、`package.json`。

## What Went Wrong

1. **发布验证命令没有保持失败传播。** 前序 build 失败没有阻止后序 pack，最后一个成功命令覆盖了整条命令的退出状态。
2. **把“存在 dist”误当成“dist 属于当前版本”。** 版本已在 `package.json` 更新，但旧构建目录仍可正常执行和打包；文件存在、命令可运行都不能证明它来自本次源码与版本。
3. **一开始容易按 workspace 依赖关系扩大版本范围。** CLI 使用 core，并不等于 core 也应发布新版本；core 是 private 且被 CLI bundle，真正面向 npm 用户的变更所有者是 `@tool-bridge/cli`。

## Root Cause

- shell 命令链的成功条件没有被显式建模，默认把最后一个进程的退出码当成整条验证链的结论。
- 发布前检查偏向 manifest 与 tarball 文件清单，缺少“先重建，再直接执行新产物读取版本”的来源证明。
- 版本 bump 范围若按源码依赖图判断，会混淆内部实现依赖与 public artifact ownership；真正需要发布的是其外部契约或可消费产物发生变化的包。

## What Worked

- registry 查询先确认 `@tool-bridge/cli` 的 `latest` 仍为 0.7.0，使 0.8.0 的 minor bump 有明确基线，并与本轮参数契约和迁移行为的用户可见变化匹配。
- 发布范围最终按 public artifact ownership 收敛：只提升 CLI 到 0.8.0；core 保持 private、不单独 bump，SDK、Gateway、Dashboard、Server 等其他 public 包没有对应产物变化，也不跟随 bump。
- 修正后的命令用 `&&` 串联 build、产物版本检查和 dry-run pack，任何一环失败都会阻止后续步骤，最终成功退出码能够代表整条证据链。
- `node dist/index.js --version` 从重建后的真实入口报告 0.8.0，避免只读取 `package.json` 自证；`npm pack --dry-run --json` 同时证明发布白名单没有夹带源码、测试或工作区文件。
- 发布准备保持边界清晰：当前只完成版本和发布物验证，尚未打 tag、尚未发布，不能把 0.8.0 描述为 registry 已可用。

## Missing Docs or Signals

- npm 发布指南强调了 dry-run 和 tarball 清单，但还应明确：发布物验证必须从干净或明确重建的 `dist` 开始，并直接运行发布入口核对版本。
- 验证指南尚未明确规定多阶段发布检查必须使用 `&&`（或等价的 fail-fast 机制），也没有提醒“末尾成功会掩盖前序失败”。
- 版本范围判断可以补一条稳定规则：先列 public artifacts 及其所有者，再判断哪些外部契约或包内容发生变化；不要按 workspace 依赖传递式 bump。

## Promotion Candidates

1. **发布命令链必须 fail fast：**build、产物检查、pack 等阶段用 `&&` 或脚本级严格错误处理连接，不接受只看最后一个命令退出码的证据。
2. **版本证据来自重建产物：**manifest version 只是输入；发布前必须重建并执行实际入口（或检查生成元数据）确认产物版本，再进行 dry-run pack。
3. **按 public artifact ownership 决定 bump：**private 且被 bundle 的内部包不因实现变化自动单独 bump；其他 public 包仅在自身外部契约、依赖约束或发布内容变化时提升版本。
4. **准备、推送、tag、发布分别报告：**本地 0.8.0 tarball 验证成功不等于 commit 已推送，更不等于 tag workflow 或 npm registry 已完成。

## Task-Specific Evidence

- 全仓验证基线：`pnpm verify` 为 **1103 passed / 7 skipped**。
- npm registry 基线：`@tool-bridge/cli` 的 `latest` 为 0.7.0。
- 正确验证链：`./node_modules/.bin/tsup && node dist/index.js --version && npm pack --dry-run --json`；重建入口报告 0.8.0。
- dry-run tarball 文件仅有 `LICENSE`、`README.md`、`dist/index.js`、`package.json`。
- 本轮决定只将 public `@tool-bridge/cli` bump 到 0.8.0；core 不发布，其他 public 包不 bump。当前未打 tag、未发布。

## Follow-up

- 后续 commit/push 后仍需把 Git 提交状态与发布状态分开核对；只有单独推送 `cli-v0.8.0` tag、发布 workflow 成功且 `npm view @tool-bridge/cli version` 返回 0.8.0，才能宣称 npm 发布完成。
- recorder 可将 fail-fast 命令链、重建产物版本核对和 public artifact ownership 三条规则提炼进 npm 发布与验证指南。
